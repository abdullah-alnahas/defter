1. Big picture

defter is a tiny three-layer system: a browser running a single-page app, a Rust web server that hands it data, and a folder of markdown files instead of a database.

Compared to what you know: swap FastAPI for Actix (Rust web framework), swap a React/Vue SPA for a Svelte SPA, and swap a Postgres/SQLite DB for a folder of markdown files. The shape is the same.

2. The layers

2.1 Browser runtime

Loads index.html, runs the compiled Svelte JS, paints pixels. Same role as a browser running a React or vanilla-JS app.

Responsibilities: rendering, listening for clicks, keeping the URL bar in sync, sending fetch() calls.

2.2 Frontend — Svelte 5 SPA frontend

A single-page application: one HTML file, one JS bundle. Navigation never reloads the page — JS rewrites the URL and swaps which component is shown.

Responsibilities

  • Decide which view to show based on URL (/ → index, /p/<slug> → page).
  • Call the backend with fetch('/api/...').
  • Convert markdown text to HTML (marked library).
  • Wrap every page in the mandatory basmalah/hamd/salawat layout (Page.svelte).
What's a "Svelte component" in one paragraph?

A .svelte file is a self-contained piece of UI. It looks like an HTML file with <script>, markup, and <style> sections. At build time the Svelte compiler turns it into plain JavaScript that creates and updates DOM nodes. Think Jinja templates that also carry their own behaviour — except instead of running on the server they run in the browser.

2.3 Backend — Actix-web (Rust) backend

An HTTP server written in Rust. Two jobs:

  1. Serve the SPA — hand back frontend/dist/index.html + the JS/CSS bundle for any non-API URL.
  2. Serve the data API — two endpoints that read markdown files and return JSON.

If you've written FastAPI handlers, the mental model is identical: function decorated with a route, returns a response. The only differences are the language and that Rust gives you compile-time guarantees.

"Why Rust if it's a personal site?"

Honest answer: low memory, single static binary, fast cold start, owner enjoys it. None of those are strictly required at this scale — a FastAPI version would work fine. The architecture below would barely change.

2.4 "Database" — the content/ folder content

There is no database. The backend reads content/*.md from disk on every request. Each file is a row.

Each markdown file starts with a small block of metadata (the frontmatter) between +++ markers, then the body:

+++
title = "On reading slowly"
lang  = "en"
dir   = "ltr"
date  = "2026-05-04"
+++

Reading slowly is not a failure of speed…

The filename (minus .md) is the page's slug — the URL piece.

FastAPI analogy: imagine a SQLAlchemy model Page(slug, title, lang, dir, date, body). Now imagine your "DB" is literally os.listdir("content/"). That's it.

Why split backend/ and frontend/? Different toolchains. Rust uses cargo; the Svelte side uses bun/npm/Vite. Keeping them in sibling folders means each side runs cargo … or bun … without stepping on the other.

Why lib/ vs pages/? Pure convention. pages/ = top-level views you can land on by URL. lib/ = reusable building blocks. (Same idea as app/views.py vs app/components.py.)

4. A concrete request lifecycle

Scenario: the user is on the home page and clicks the link "On reading slowly". Here is everything that happens, end to end.

1
FEClick. The user clicks <a href="/p/on-reading-slowly"> rendered by IndexPage.svelte. The element is actually a <Link> component that intercepts the click so the browser doesn't do a full page reload.
2
FERoute change. Link calls router.navigate('/p/on-reading-slowly'), which calls history.pushState (browser URL changes, no reload) and updates a reactive path variable. App.svelte sees the new path and renders <PageView slug="on-reading-slowly" /> instead of <IndexPage />.
3
NETHTTP call. PageView kicks off fetch('/api/pages/on-reading-slowly'). This is the same fetch() you'd write in vanilla JS — nothing Svelte-specific about it.
4
BEHandler matches. Actix routes the request to get_page(slug: web::Path<String>). This is the equivalent of FastAPI's @app.get("/api/pages/{slug}") async def get_page(slug: str):.
5
BEValidate. The slug is checked (no /, no ..) so a clever URL can't read random files on the server.
6
FSRead file. page::load_one opens content/on-reading-slowly.md and reads the bytes.
7
BEParse. The bytes are split at +++ markers; the top half is parsed as TOML (gives title, lang, dir, date) and the bottom half is kept as the raw markdown body.
8
BESerialize. The parsed struct is turned into JSON automatically by serde (Rust's serialization library — think pydantic's .model_dump_json()).
9
NETResponse. 200 OK, Content-Type: application/json, body = 
{ "slug":"on-reading-slowly", "title":"On reading slowly",
  "lang":"en", "dir":"ltr", "date":"2026-05-04",
  "body":"Reading slowly is not a failure of speed…" }
10
FERender. PageView awaits the promise, passes the body through marked.parse() (markdown → HTML string), and mounts it inside the <Page> wrapper, which itself renders the opening basmalah, the page content, and the closing salawat.

FastAPI mirror: the only step that has no FastAPI equivalent is the build-time compilation of Svelte components. Everything else maps line-for-line: a request hits a typed handler, a model is serialized, JSON goes back over the wire.

User session / login Does not exist yet. —
Server-side cache Does not exist yet — every request re-reads the file. You'd add Redis or @lru_cache.

The backend is stateless between requests: it holds no in-memory data about pages or users. Every API call is a fresh "read the folder" operation. This is deliberate — fewer moving parts.

6. Where authentication happens

Currently: nowhere. defter is a read-only public site. There is no login, no session, no admin. Anyone can read the API; nobody can write through it.

The intended future is an editor (per concept.txt). When that lands, the natural place to plug auth in is:

  • A new POST/PUT /api/pages/<slug> handler.
  • An Actix middleware in main.rs that checks an auth header before the handler runs. Same idea as a FastAPI Depends(get_current_user) dependency.
  • Frontend stores a token (cookie or localStorage) and attaches it to fetch() calls.

7. How frontend and backend communicate

One protocol: HTTP. Two endpoints today:

Method + pathWhat it returnsWho calls it
GET /api/pagesJSON array of metadata (no bodies), sorted newest firstIndexPage.svelte
GET /api/pages/{slug}JSON object with metadata + markdown bodyPageView.svelte
GET /anything-elseThe SPA's index.htmlThe browser, on first load or refresh

That last row is the only piece that looks unusual. It's called the SPA fallback: if the URL doesn't match a static file or an API route, return index.html so the Svelte router can take over. Without it, refreshing on /p/on-reading-slowly would 404 (because there is no such file on disk — that path exists only in JS).

8. Small annotated snippets

8.1 The whole backend, in 30 lines (paraphrased)

#[get("/api/pages")]                       // like @app.get(...)
async fn list_pages() -> impl Responder {
    match page::load_all(&content_dir()) {     // read folder, parse each .md
        Ok(pages) => HttpResponse::Ok().json(pages),
        Err(_)    => HttpResponse::InternalServerError().finish(),
    }
}

#[get("/api/pages/{slug}")]
async fn get_page(slug: web::Path<String>) -> impl Responder {
    match page::load_one(&content_dir(), &slug) {
        Ok(p)                       => HttpResponse::Ok().json(p),
        Err(page::PageError::NotFound) => HttpResponse::NotFound().finish(),
        Err(_)                       => HttpResponse::InternalServerError().finish(),
    }
}

#[actix_web::main]
async fn main() -> std::io::Result<()> {
    HttpServer::new(|| {
        App::new()
            .service(list_pages)               // register the two routes
            .service(get_page)
            .service(Files::new("/", &dist)    // serve the Svelte build
                       .index_file("index.html")
                       .default_handler(...))  // SPA fallback
    })
    .bind("127.0.0.1:8787")?.run().await
}
Rust idioms to recognize in the snippet above
  • #[get("…")] — an attribute macro. Same role as FastAPI's @app.get("...") decorator.
  • match … { Ok(x) => …, Err(e) => … } — pattern matching on a Result. Python equivalent: try/except, but explicit in the type signature.
  • impl Responder — "this function returns something that can be turned into an HTTP response." Similar to FastAPI letting you return a dict, a Pydantic model, or a Response.
  • async fn — same as Python's async def.
  • & in &slug — "borrow this value, don't take ownership of it." Unique to Rust; treat it as "pass by reference" for now.

8.2 The frontmatter parser (the heart of page.rs)

fn split_frontmatter(src: &str) -> Result<(&str, &str), PageError> {
    // 1. file must start with "+++\n"
    let rest = src.strip_prefix("+++")?.strip_prefix('\n')?;
    // 2. find the closing "\n+++"
    let close = rest.find("\n+++")?;
    let fm   = &rest[..close];                       // metadata block
    let body = rest[close + 4..].trim_start_matches('\n'); // markdown body
    Ok((fm, body))
}

That's the entire format. No regex, no library, no schema. The TOML block is then parsed with the toml crate into a typed Frontmatter struct.

8.3 The frontend router, in full

let _path = $state(window.location.pathname);   // reactive variable

window.addEventListener('popstate', () => {
  _path = window.location.pathname;                  // back/forward buttons
});

export const router = {
  get path() { return _path; },
  navigate(to) {
    if (to === _path) return;
    history.pushState({}, '', to);                   // change URL, no reload
    _path = to;                                      // triggers re-render
  },
};

That's the whole router. No library. The magic word is $state(...): Svelte 5's way to say "this variable is reactive — any component that reads it will re-render when it changes." Mentally: useState from React but as a language feature.

  • $props() — declares which props this component accepts. Equivalent to a React function component's argument destructuring.
  • {@render children?.()} — "render whatever the parent component nested inside me." Equivalent to React's {children} or Vue's default <slot/>.
  • {dir} {lang} — Svelte shorthand for dir={dir} lang={lang}.

11. Why these choices

Why a SPA at all, instead of plain server-rendered HTML?

The site's concept calls for in-page interactivity (hidden-on-hover navbar, TL;DR buttons, future executable code blocks). Those want a JS runtime anyway. Once you've decided you need JS, a SPA gets you free client-side navigation (no full reloads between pages → calm reading experience).

Why filesystem instead of a database?

Posts are written in markdown anyway. A folder is the simplest possible store: it's diff-able in git, editable in any text editor, and survives without a running daemon. The price is no full-text search and no concurrent writes — neither matters for a one-author blog.

Why a mandatory wrapper component (Page.svelte)?

The owner wants every page on the site to open with basmalah/hamd/salawat and close with salawat/hamd. Putting it in a wrapper makes it impossible to forget: if you don't go through <Page>, you can't render content.

Why TOML frontmatter instead of YAML?

Smaller surface area, no indentation traps, and Rust's toml crate is mature. Functionally interchangeable with YAML for this metadata.

12. Glossary

actix-web
A Rust HTTP framework. Same niche as FastAPI / Flask / Express.
async / .await
Same idea as Python async/await. A function marked async returns a future; .await pauses until it resolves.
Bun
A drop-in replacement for Node + npm. Used here only to install JS dependencies and run Vite.
cargo
Rust's package manager and build tool. Combines what pip + build + pytest + tox do.
crate
The Rust word for a package/library. Listed in Cargo.toml, fetched on cargo build.
derive (#[derive(Serialize)])
A Rust attribute that auto-generates boilerplate for a struct — here, the code that turns the struct into JSON.
frontmatter
The metadata block at the top of a content file, fenced by +++ (TOML) or --- (YAML). Borrowed from static-site generators.
history API
Browser API (history.pushState, popstate) that lets JS change the URL without reloading the page. Foundation of every SPA router.
marked
A small JavaScript markdown parser. Turns "# hi" into "<h1>hi</h1>".
middleware (Actix)
A function that wraps every request before/after the handler runs. Same role as FastAPI middleware.
props
Inputs passed into a component from its parent. Same word, same meaning as in React/Vue.
Result<T, E>
Rust's "either a success value of type T or an error of type E." Python equivalent: try/except, but the success/error split is explicit in the type signature.
runes ($state, $props, $derived)
Svelte 5's reactivity primitives. They look like function calls but are recognized by the compiler.
serde
Rust's de-facto serialization library. Converts structs ↔ JSON, TOML, etc. Mental model: Pydantic + json.dumps in one.
slug
The URL-safe identifier for a page. Here, the filename minus .md.
SPA
Single-Page Application. The server hands over one HTML file once; thereafter JS swaps content in place.
SPA fallback
The trick of returning index.html for any unknown URL so the JS router can handle it. Needed because the JS doesn't run if the server returns 404 first.
Svelte
A JS UI framework whose distinguishing feature is that components compile down to small, framework-less JS. No virtual DOM.
TOML
A small config-file format. Used for both Rust's Cargo.toml and the page frontmatter.
Vite
The build tool that compiles Svelte components into a JS bundle. Equivalent role to webpack or Parcel.
