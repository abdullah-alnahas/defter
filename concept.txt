# Things I want my would-be personal website to have, insha'Allah

## Home page
Blog view: Picked (featured) + All (full list). No separate "Recent" section — recency is implicit in the All ordering.

## Navigation
- Persistent left-margin aside (always visible). Links: Blog, Featured, About, GitHub, LinkedIn, RSS.
- No top navbar. The aside is the only nav surface.
- Skip-link at top for keyboard users.

## Page concept
Everything on the website is conceptually a "page." A page can be:
- A blog post
- An app (with backend and frontend, or frontend only, that serves something)
- The About page

Blog pages follow a standardized design. App pages and the About page are arbitrary code. The About page is essay-only: voice-first, four short paragraphs, contact block, and a single "Download CV (PDF)" disclosure. The CV does **not** live as inline content on the about page — instead the disclosure lets the visitor pick a palette + light/dark variant, then opens `/p/cv/print` in a new tab. That route renders a chrome-free, themed CV and auto-fires `window.print()`, so the visitor saves a PDF via the system print dialog. The printed CV deliberately omits the site's opening basmalah / hamd / salawat and closing Ayah / Ibrahimi salawat (those framings belong to the website surface, not to a CV artifact). CV content lives in `src/lib/cv-data.js` as a single source. `/p/cv` is preserved as a 308 redirect to `/p/about` for backlink compatibility.

## Tech stack
- SvelteKit + Svelte 5 (runes), `adapter-static` — pure prerender, no server runtime in production. One toolchain, one language.
- Markdown pipeline: `marked` + `marked-footnote` + `marked-gfm-heading-id`, executed at build time inside `+page.server.js` loaders.
- Bun for install + script runner; Vite under the hood (SvelteKit's bundler).
- Future "app pages" (concept below) may bring back a backend per-app (Rust, Python, whatever fits), but the blog/CV/static surface does not need one.

## Page editor features
For any page I create in the website editor:
- Arbitrary JS code in a block — for example, to demo an idea inline. (Maybe call it an "executed code block" rather than just "code block"? Sounds a bit mouthy though.)
- Several themes, each with a dark and a light variant.
- Extremely minimal looking but rich. Examples:
  - No top navbar — nav lives quietly in the left margin, always visible but dim.
  - Each published page has a TL;DR trigger styled as a subtle button next to the title — dim by default, brightens on hover, clicks pin a TL;DR margin note like a sidenote.
  - Plenty of negative space.
  - Best UI/UX practices.
  - Optimized for reading, calm, relaxation, eye comfort, no clutter, no distractors, and a single focus point for the user.

## Page structure
Each page necessarily starts with: basmalah, hamd, salawat — all rendered with diacritics. Basmalah and hamd use the Quran font (UthmanicHafs), the opening salawat uses the regular Arabic stack.

Each page necessarily ends with:
1. The Ayah (rendered with the Quran font, UthmanicHafs — ayah separators are the proper U+06DD rosette with verse number, not asterisks):

   ﴿سُبْحَانَ رَبِّكَ رَبِّ الْعِزَّةِ عَمَّا يَصِفُونَ ۝١٨٠ وَسَلَامٌ عَلَى الْمُرْسَلِينَ ۝١٨١ وَالْحَمْدُ لِلَّهِ رَبِّ الْعَالَمِينَ ۝١٨٢﴾

2. The Ibrahimi salawat, with the honorific "سيدنا" placed before each mention of Prophet Muhammad ﷺ and Prophet Ibrahim عليه السلام:

   اللّهُمَّ صَلِّ على سيِّدِنا مُحمَّدٍ وعلى آلِ سيِّدِنا مُحمَّدٍ، كَما صَلَّيْتَ على سيِّدِنا إبراهيمَ وعلى آلِ سيِّدِنا إبراهيمَ، إنَّكَ حَميدٌ مَجيدٌ. اللّهُمَّ بارِكْ على سيِّدِنا مُحمَّدٍ وعلى آلِ سيِّدِنا مُحمَّدٍ، كَما بارَكْتَ على سيِّدِنا إبراهيمَ وعلى آلِ سيِّدِنا إبراهيمَ، إنَّكَ حَميدٌ مَجيدٌ.

## Sidenotes + TL;DR (one mental model, one surface)
Interactive margin notes that live invisibly in the right margin column and reveal on demand.

- **Each footnote reference** spawns its own margin slot at the ref's vertical position (float-down on collision).
- **Hover ref or hover note** → reveal (opacity 0 → 1).
- **Click ref** → pin (stays revealed until clicked again or `Esc`).
- **Same label opened twice** → two separate slots (per-occurrence identity).
- **TL;DR** uses the same surface and same interaction model: a small dim "TL;DR" (or "خلاصة" on Arabic pages) button next to the title spawns a margin note. Hover the button → preview. Click → pin. The trigger label, margin-note label, and endnote-section heading all localize with the article's `lang`.
- **Pin/Unpin all** pin-icon toggle lives in the theme bar (top-end) and only surfaces when the page has sidenotes — flips every sidenote (and TL;DR) at once.
- **Endnote list** is always present after the article body for assistive tech, print, and feed readers.
- **Narrow viewports** (≤ 56rem) collapse the right margin column to flow below the body — notes still hidden until hover/tap of their ref.

## Typography
- **Body + nav:** Montserrat (self-hosted woff2, 400 + 500 + 600, latin + latin-ext subsets, `font-display: swap` so the web font replaces the fallback whenever it arrives — `optional` caused permanent fallback on cold-cache visits). The 400/500/600 weights are `<link rel="preload">`'d so they typically land inside the first-paint window; CLS stays 0. Single Latin family — no separate UI font.
- **Quran:** UthmanicHafs (subset to Arabic blocks; auto-wraps Arabic-Indic digit sequences in the ayah-marker rosette via its own GSUB tables — no explicit U+06DD needed).
- **Regular Arabic:** Noto Naskh Arabic (self-hosted), fallback Amiri / Scheherazade New.

## Layout
- Symmetric 3-column grid: equal-width margin / body (capped at `--measure`) / equal-width margin. Body is centred. Sidenotes live in the right margin column.
- Nav aside: fixed top-start (left).
- Theme bar (pin-all + palette + variant): fixed top-end (right). The palette button opens a popover containing a **radial dial** — dots on the circumference of an invisible circle (paper at the top, walking clockwise through a hue-ordered sequence: sepia → cream → ember → amber → forest → slate → ink → harbor → midnight → magenta → rose → mono). The active palette is mirrored as a centre swatch + label; on its own dot it shows as an inset pip so the dot stays glued to the radius. Selecting a palette switches in place; no blind cycling required.
- Back-to-top button: fixed bottom-end. Always visible — arrow + "Back to top" label.
- Text selection (`::selection`) reads `--accent` / `--bg` so the highlight tracks the active palette.

## Languages and direction
Blogs should support Arabic, Turkish, and English by default. A title, paragraph, or other element may use any mix of these. The website should support both RTL and LTR. Insha'Allah!

## Blog post elements

- Arbitrary executed code block
- Ayah
- Hadith

## Quality bar
The website must achieve a Lighthouse score of 100 in all four categories — Performance, Accessibility, Best Practices, SEO — on every page, on **desktop**. Mobile target is 100 across A11y/BP/SEO and ≥ 99 on Performance (the inlined hydration bundle plus Montserrat preload puts FCP at the 1.0–1.4s range on simulated Slow 4G — within "good" Core Web Vitals but rounded just below a perfect score). Treat 100×4 desktop as a hard constraint. Concrete implications:
- Static prerender per page so initial HTML carries full content, correct `<title>`, `<meta description>`, `<html lang>` and `dir` — no blank SPA shell.
- Every hover-revealed control (TL;DR trigger, sidenote reveal) must have a keyboard-equivalent (focus-visible, skip links).
- Color tokens must satisfy WCAG AA contrast (≥ 4.5:1 body, ≥ 3:1 large text and UI components) across every theme variant.
- Images carry explicit width/height, modern format, lazy loading below the fold. Same for executable iframes — explicit `width`/`height` attributes prevent CLS.
- Per-page Open Graph / Twitter card meta; sitemap + robots.txt served at root.
- No console errors, no deprecated APIs, no render-blocking third-party scripts.
- `font-display: swap` on web fonts (with preload for the body weights) so first paint never blocks on the font network round-trip and cold-cache visitors still receive the web font when it arrives. CLS verified 0 across measured routes.
