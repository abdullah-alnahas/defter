# Things I want my would-be personal website to have, insha'Allah

## Home page
Not entirely decided yet. By default, it could be the blog view, showing: picked blogs, recent blogs, then the full list of all blogs.

## Navbar
- CV
- Blog
- GitHub
- LinkedIn
- RSS feed (I don't yet know what it is or how it works, lol)
- Featured — pages I want to feature. The view would be a single centered column containing a card for each project page. Each card would show a brief description, an external link if applicable, and a link to the page on my website.

## Page concept
Everything on the website is conceptually a "page." A page can be:
- A blog post
- An app (with backend and frontend, or frontend only, that serves something)
- The CV

Blog pages would follow a standardized design. App pages and the CV page would be arbitrary code.

## Tech stack
- SvelteKit + Svelte 5 (runes), `adapter-static` — pure prerender, no server runtime in production. One toolchain, one language.
- Markdown pipeline: `marked` + `marked-footnote` + `marked-gfm-heading-id`, executed at build time inside `+page.server.js` loaders.
- Bun for install + script runner; Vite under the hood (SvelteKit's bundler).
- Future "app pages" (concept below) may bring back a backend per-app (Rust, Python, whatever fits), but the blog/CV/static surface does not need one.

## Page editor features
For any page I create in the website editor:
- Arbitrary JS code in a block — for example, to demo an idea inline. (Maybe call it an "executed code block" rather than just "code block"? Sounds a bit mouthy though.)
- Several themes, each with a dark and a light variant.
- Extremely minimal looking but rich. Examples:
  - The top navbar is hidden by default and appears on hover.
  - A hidden sections navigator on the left, also revealed on hover.
  - Each published page has a TL;DR button that is hidden by default and appears on hover.
  - Plenty of negative space.
  - Best UI/UX practices.
  - Optimized for reading, calm, relaxation, eye comfort, no clutter, no distractors, and a single focus point for the user.

## Footnotes
Interactive footnotes. Two interactions, one design:

- **Hover (or keyboard focus) a footnote reference** → an ephemeral preview popover floats next to the ref. Only one preview at a time; moving to another ref replaces it; moving away dismisses it.
- **Click a reference** → the note is *pinned* into a shared sticky stack in the right margin (the "margin column"). Pinned notes accumulate, sorted by their reference's vertical position in the document, so the stack reads top-to-bottom in document order. Each pin shows an unpin × control; the stack header shows "Pinned (N)" and a "Clear all" button. Pressing `Esc` clears all pins. Pins live in memory only — fresh tab = empty stack.
- **Same label opened twice** → counts as two separate pins (per-occurrence identity).
- **Endnote list** is always present after the article body, regardless of pins. This is the canonical reading affordance for assistive tech, print, and feed readers.
- **Narrow viewports (no room for a margin column)** → no preview popover; clicking the ref toggles an inline expansion immediately under the ref instead.

## Languages and direction
Blogs should support Arabic, Turkish, and English by default. A title, paragraph, or other element may use any mix of these. The website should support both RTL and LTR. Insha'Allah!

## Quality bar
The website must achieve a Lighthouse score of 100 in all four categories — Performance, Accessibility, Best Practices, SEO — on every page. Treat this as a hard constraint, not an aspiration. Concrete implications:
- Static prerender per page so initial HTML carries full content, correct `<title>`, `<meta description>`, `<html lang>` and `dir` — no blank SPA shell.
- Every hover-revealed control (navbar, section nav, TL;DR) must have a keyboard-equivalent (focus-visible, skip links).
- Color tokens must satisfy WCAG AA contrast (≥ 4.5:1 body, ≥ 3:1 large text and UI components) across every theme variant.
- Images carry explicit width/height, modern format, lazy loading below the fold.
- Per-page Open Graph / Twitter card meta; sitemap + robots.txt served at root.
- No console errors, no deprecated APIs, no render-blocking third-party scripts.
